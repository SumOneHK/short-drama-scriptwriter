# AI漫剧策划与剧本生成

本文是 skill 的入口路由。详细规则、产物等级、状态机、写作方法都在 `references/`，按 tier 标记按需加载。

## 适用边界

只服务 `AI 制作的竖屏剧 / 漫剧`，不服务真人拍摄短剧。

固定边界：
- 策划层用中文；出海项目只有 `剧本对白 / 屏幕字 / 必要制作语种字段` 用目标语，未指定时美国英语首版默认英文
- 平台口径：国内写 `抖音`，出海写 `TikTok`；目标市场单独锁（`国内 / 美国英语首版 / 泛欧美 / 东南亚 / 拉美 / 指定地区`）
- 所有交付文件统一 Markdown；最终剧本正文统一 `场次执行稿`
- 本 skill 只到 `场次执行稿` 这一层；分镜、视频提示词、配音脚本、音效表交下游流水线，本 skill 仅在 `production-enhanced` 包里补齐资产命名、场景/道具卡等对接资料
- AI 制作能力**不作为题材降级理由**；奇幻、异世界、末日、规则世界、大场面都可以入选，选题只判断视觉锚点 / 角色识别 / 世界规则 / 情绪引擎是否清楚
- 用户没有明确确认创作方向前，`/立项` 只在对话中做调研、比较和候选汇总；不创建项目目录、不落任何立项文件、不进入 `T02`

## 核心目标

产出有吸引力、可交付、能让观众继续看的剧本，不是完成模板。

- 立项能说明观众为什么点开、为什么追下去
- 策划能说明人物欲望、关系张力、反派压迫和阶段回报如何持续升级
- 分集大纲能说明每集为什么必须存在
- 正文能在场面里呈现开场抓力、可见冲突、动作推进、情绪变化和尾钩拉力
- 交付包能让制作方直接执行，不需要反查工作稿
- 全链路本地化深度成立——题材 / 矛盾 / 压迫机制 / 爽点 / 视觉锚点 / 对白都长在目标市场真实社会纹理里，不是直译版

商业化原则：
- 商业化默认**后置**；不用 IAA / IAP / 付费阶梯倒推选题或大纲
- `卡点` 在剧作层先理解为"让观众继续看的具体戏剧问题"，不是默认广告位或付费点
- 用户明确要求投流、IAA、IAP、平台付费或商业化大纲时，再读取 [references/24-卡点设计.md](references/24-卡点设计.md) 追加商业字段

本地化贯穿全流程，详细方法分布：跨市场 trope 重写矩阵在 [references/02-策划细则.md](references/02-策划细则.md)；选题词库在 [references/15-国内选题与设定词库.md](references/15-国内选题与设定词库.md) / [references/16-美国英语市场选题与设定词库.md](references/16-美国英语市场选题与设定词库.md)；对白本地化在 [references/12-单集写作协议.md 第七章](references/12-单集写作协议.md)；本地化深度门槛在 [references/13-质量检查表.md](references/13-质量检查表.md)。

## 主流程

四段：`立项与选题 → 剧本圣经设计 → 分集剧本创作 → 交付整理`
剧作主链：`创意 → 梗概 → 大纲/情节点 → 剧本初稿`

T01-T18 任务包、产物等级、创建时机的**单一真相源**是 [references/21-产物矩阵与创建时机.md](references/21-产物矩阵与创建时机.md)；本文不重复维护。
状态机、`currentStep`、门禁节点和回退规则的**单一真相源**是 [references/17-状态流转与阶段门禁.md](references/17-状态流转与阶段门禁.md)。

执行原则：
- 立项阶段先做市场判断 / 选题沟通 / 目标确定；用户未明确确认最终方向和必要立项信息前，只停留在对话讨论
- 选题策略不默认推高概念——先判断 `主流稳态 / 类型微创新 / AI-native 视觉型 / 高概念创新` 哪种适合用户，再给候选；高概念只是可选策略之一
- `T05-T07`（世界观 / 角色 / 场景道具）是同一轮故事基底包，按 [references/02-策划细则.md](references/02-策划细则.md) 的"同轮迭代五步法"执行；不是三个孤立填表
- `T08-T09` 稳定后再校准 `T05-T07`，不提前写百科式世界观或资产大全
- `分集大纲.md` 是正文前**硬门槛**；未全量完成且未通过 `大纲质检`，不进入 `/剧本`
- `/剧本` 默认按批次写；每批先写 `1-2` 集代表样稿，通过后再铺满本批
- `/复盘` 是显式准入动作，不替代前面阶段的创作判断
- `/交付` 只整合正式源文件，不在交付阶段重写剧情

如果用户已有创意 / 梗概 / 大纲，从对应节点切入；不要为流程完整反向补一堆不影响梗概和初稿的设定文件。

## 状态与门禁

- 用户确认进入 `T02` 时初始化 `项目状态.json` 和 `90-内部工作稿/质检检查点.md`
- 阶段门禁以这两份文件为单一真相源
- 当前上下文里的自检只算建议，**不算门禁放行**
- 独立复检的判定标准、可机器验证证据、运行环境（Claude Code / Codex / 普通会话）适配方法见 [references/18-过程质检与独立复检.md](references/18-过程质检与独立复检.md)

- 上游文件实质改动后，下游 `qcStatus` 必须回退；详见 17 的 `changeControl`

## 模板字段三档分级（MUST / SHOULD / MAY）

正式模板字段按 RFC 2119 风格分三档：

- `MUST`（必填）：缺字段视为流程不完整；独立质检判 `需修改`
- `SHOULD`（默认填）：跳过须在阶段回复或质检检查点写明跳过原因
- `MAY`（条件触发）：常规项目可不填；只在命中触发条件时才需要

反作弊红线：
- 不允许把跳过的字段填成 `待定 / 暂略 / 后补`
- SHOULD / MAY 字段在条件不满足时填了**也不加分**
- 字段填得满 ≠ 完成度高；评估完成度只看 MUST 字段是否实质填写

详细判定（含字段命中条件示例和质检写法）见 [references/19-质量门禁.md](references/19-质量门禁.md) 和 [references/18-过程质检与独立复检.md](references/18-过程质检与独立复检.md) 的字段分档审查段。

## 强规则

- 题材未锁定前，不进入 `/策划`
- 没有全量 `分集大纲.md` 且未通过 `大纲质检`，不进入 `/剧本`
- 剧本正文必须基于已通过的大纲生成
- 每批正文未过 `剧本质检`，不进入下一批
- `项目状态.json` 和 `质检检查点.md` 缺失时，不能把项目视为真正过门禁
- 同一上下文里的自检不能写成门禁放行；没有独立复检证据时 `qcStatus` 保持 `未检查` 或既有状态
- 海外项目策划层统一中文，英文只进入剧本对白和必要制作字段
- `完整剧本总稿.md` 只汇总正式分集 `场次正文`，不包含 `本集创作简卡 / 本集回填 / 复盘 / 检查单 / 批次写作单`
- `剧本圣经.md` 只在交付阶段生成，只整合正式策划内容；如果汇总了场景/道具章节，仍保留独立场景/道具卡作为制作资产卡 source-of-truth

## 脚本

脚本只做交付汇总和单集硬阈值机械检查，**不做戏剧判断**。质检放行必须由模型按 `references/18` 读实际稿件完成。

```bash
# 交付汇总（默认验门禁；--draft 跳过校验生成草稿；--check-mode 只比 mtime）
python3 scripts/build_full_script.py projects/<project-slug>

# 单集硬阈值检查（叠加屏幕字 / 画内可读性 / 规则词频次 / 前置比例 / 有效行数）
python3 scripts/check_episode.py \
    --regulation-words "命格,暗文,命债,见证人" \
    projects/<project-slug>/02-剧本/第001集.md
```

脚本输出可作为独立复检的"硬阈值证据附件"，但**不替代独立复检的戏剧判断**。详见 [references/18-过程质检与独立复检.md](references/18-过程质检与独立复检.md) 的"可机器验证的 clean-context 证据"。
