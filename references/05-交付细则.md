# 交付层细则

本文负责 `/交付` 的入口判断、标准交付包和交付前预检；具体产物是否创建以 [21-产物矩阵与创建时机.md](21-产物矩阵与创建时机.md) 为准。制作增强包的资产命名、海外对白 source-of-truth 和渠道特化细则只在 [14-制作交付整理.md](14-制作交付整理.md) 维护。

## 目标

- 把已完成的剧本整理成制作方可执行的交付包
- 不在交付阶段重写剧情

## 输出

默认先交 `标准交付包`：
- `完整剧本总稿.md`
- `剧本圣经.md`
- `交付清单.md`

命中 `production-enhanced` 时，再按 [21-产物矩阵与创建时机.md](21-产物矩阵与创建时机.md) 追加制作增强产物，并按 [14-制作交付整理.md](14-制作交付整理.md) 执行制作字段、资产命名和市场特化规则。

## 规则

- 用户只说“导出交付”时，默认先交 `标准交付包`
- 用户明确说“给制作方 / 制作版 / 分镜执行 / 海外制作版”时，把 `项目状态.json.deliveryMode` 改为 `production-enhanced`，再补 `制作增强包`
- 项目本身为 `资产重 / 海外 / 强设定` 时，进入正式制作交付前默认也应使用 `production-enhanced`；如果只交标准包，需要在 `质检检查点.md` 记录风险接受理由
- 海外项目的最终制作语种示例统一使用英文标点
- 海外项目对白的 source of truth 是 `02-剧本/第XXX集.md` 场次执行稿里的英文对白；分场表 / 制作理解稿 / 制作交付参数表 等所有派生制作产物都从 source 派生，不允许反向修改 source；详见 [14-制作交付整理.md](14-制作交付整理.md) 的 `海外对白 source-of-truth 锁口径`
- 最终交付包不能保留 `待补充 / 待执行` 这类占位语
- `完整剧本总稿.md` 必须从已完成单集剧本汇总
- `剧本圣经.md` 标准交付默认汇总 `卖点与项目定位 / 项目设定 / 故事梗概 / 故事大纲 / 故事节拍表 / 阶段大纲（若有） / 世界观设定 / 人物小传`
- `剧本圣经.md` 进入 `production-enhanced` 时，还应纳入 `制作理解稿 / 制作交付参数表 / 信息流投放素材切片说明（抖音国内）/ 场景参考卡 / 道具设定卡`，形成单册可读的制作总交底
- `剧本圣经.md` 只有进入 `production-enhanced` 或交付层已有正式 `场景参考卡 / 道具设定卡` 时，才纳入场景和道具章节
- `03-交付/场景参考卡.md` 与 `03-交付/道具设定卡.md` 是制作资产卡的 source of truth；`剧本圣经.md` 中的同名章节只是汇总副本，方便单册阅读，不替代独立资产卡
- 更新独立场景 / 道具卡后，必须重新运行 `build_full_script.py`，让 `剧本圣经.md` 的汇总副本同步刷新
- 新增或补齐交付文件后，重新运行 `build_full_script.py`，刷新 `完整剧本总稿.md / 剧本圣经.md / 交付清单.md`
- 如果 `01-策划/` 已有工作版 `场景参考卡 / 道具设定卡`，交付阶段要在此基础上补全成对制作方可直接使用的最终版，再落到 `03-交付/`
- `人物小传.md` 是默认角色主文件，主角、核心关系位、核心对手和主要配角的最小资产锚点也写在这里
- 如果外部平台或制作方强制要求一角色一卡，再从 `人物小传.md` 临时裁出角色资产摘要，不改变默认交付结构
- `世界观设定.md` 不单独复制到交付层，但其内容必须进入 `剧本圣经.md`
- `场景参考卡.md / 道具设定卡.md` 在交付层只在 `制作增强包` 中补齐为最终版
- 不把 `剧本圣经.md` 已包含场景 / 道具章节视为删除独立资产卡的理由；制作方、分镜和视频链路优先消费独立卡，圣经负责统一阅读
- `完整剧本总稿.md` 只收录各集 `场次正文`，不收录 `本集创作简卡 / 本集回填 / 批次写作单 / 单集达标检查 / 复盘报告 / 改稿指令单`
- `剧本圣经.md` 只在交付阶段生成，只整合正式策划文件和正式场景/道具卡，不整合任何过程工作稿
- 进入 `交付整合` 前，不要求 `qcStatus.delivery = 已通过`；只有完成 `交付质检` 后，才把交付包视为正式交付

## 脚本预检说明

`build_full_script.py` 提供三种模式：
- 默认模式：完整生成交付包，先做交付预检
- `--draft`：跳过交付预检生成草稿包；草稿不得作为正式交付放行依据
- `--check-mode`：只比对交付包与源文件 mtime，不重新生成；适合补齐场景卡 / 道具卡 / 改了某集尾钩之后快速判断"交付包是否落后"
  - 检测到任何源文件 mtime 晚于交付包文件 mtime 时，stderr 列出落后文件并返回非零退出码
  - 正式交付前应至少跑一次 `--check-mode`，避免漏跑刷新

默认模式（非 `--draft`）会自动执行以下预检（不参与质检放行，只防低级错误）：

- 校验 `项目状态.json.deliveryMode`：仍为 `planning-only` 时直接报错退出
- 校验 `qcStatus.outline = 已通过`、所有 `qcStatus.scriptBatches[]` 已通过；任意一批未通过时报错并列出具体批次
- 校验 `00-立项/锁题摘要.md` 或 `项目简报.md` 存在
- 校验所有 `REQUIRED_PLANNING_FILES` 存在；总集数 ≥ 30 时额外要求 `阶段大纲.md`
- 校验剧本集数与目标总集数连贯（缺集 / 超集都报错）

增强包推荐警告（不阻塞，stderr 输出）：
- 若 `deliveryMode = standard`，但项目命中下列任一条件，脚本会打 `[WARN]`：
  - 发行平台为 `TikTok` 的出海项目（海外项目默认应使用增强包）
  - 总集数 ≥ 30（长线项目默认应使用增强包）
  - `01-策划/` 已存在 `场景参考卡.md / 道具设定卡.md` 工作版（资产重项目）
- 命中 WARN 仍坚持只交标准包时，必须在 `90-内部工作稿/质检检查点.md` 写"风险接受理由"

## 过关条件

- 交付内容和正文一致
- 参数口径明确
- 制作方不需要再反查所有中间过程文件
- 单独打开 `剧本圣经.md` 就能看见故事梗概、结构、世界和角色的完整交底
- 如果当前是 `production-enhanced`，`剧本圣经.md` 还应包含制作理解、参数口径、渠道切片说明（抖音国内命中时）、正式场景和道具交底
