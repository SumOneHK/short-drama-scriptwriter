# 复盘层细则

`/复盘` 是显式命令，产物进入 `内部工作稿`，不单独进入前台交付目录。

## 适用情况

- 所有项目
- 长线项目
- 复杂题材
- 海外项目
- 写到中段后担心疲软或设定漂移

## 检查内容

- 反查大纲和正文是否对应
- 反查人物/场景/道具是否漂移
- 反查是否出现空集、功能集、重复回报、重复钩子
- 反查 `开场承接` 是否真正落到第一场，而不是只写在任务栏
- 反查是否出现固定开场句、固定施压句、固定反应句反复复用

## 规则

- `/复盘` 是诊断工作稿，不是门禁；详见 [18-过程质检与独立复检.md 的 `复盘的角色定位`](18-过程质检与独立复检.md)
- 复盘报告里的 `结论 / 建议` 字段不能直接改写 `qcStatus`；门禁放行只看独立质检
- 复盘的价值在 `下一批怎么改 / 怎么避免重复犯错`，不在 `本节点能不能放行`
- 复盘和独立质检结论冲突时，按独立质检处理；冲突必须在 `质检检查点.md` 写明
- 它不替代 `故事梗概 / 故事大纲 / 分集大纲 / 分集剧本`
- 它是漂移排查工作稿，不是阶段放行机制
- `/复盘` 的正式放行仍按独立复检执行；当前上下文里的自评只算诊断建议。`clean-context / subagent` 的可用条件、授权要求和证据字段统一按 [18-过程质检与独立复检.md](18-过程质检与独立复检.md) 执行，不在本文件重复定义
- 所有项目都应保留这个动作；简单项目可以弱化文档存在感，但不能跳过复盘判断
- `/复盘` 的结论不能只停留在报告里；要同步回写 `项目状态.json` 和 `质检检查点.md`
- `分集追踪.md` 不要求等 `/复盘` 才第一次生成；建议在 `/策划` 复盘通过后、首批正文前初始化，`/复盘` 负责检查它有没有及时回填
- 纯策划阶段 `/复盘` 不以 `分集追踪.md` 为必需文件；一旦开始正文批次，它就应存在并持续回填
- `/复盘` 默认检查这几类内部工作稿是否各司其职：
  - `试写样场` 有没有只停留在试写，不被误当成正式正文
  - `批次写作单` 有没有只服务当前批次，不被写成第二份大纲
  - `单集达标检查` 有没有真正承担准入判断，而不是变成重复剧情
  - `分集追踪` 有没有持续累计稳定事实，而不是被写成下一批剧情计划
- 每一份复盘报告默认在同一文件里完成 `策划视角 / 编剧视角 / 导演视角` 三个 section
- 不默认拆成三份并行文件，避免质检重新做重
- 复盘报告按里程碑命名，避免不同阶段互相覆盖
- 复盘结论如果是 `否`，不进入下一批，不进入 `/交付`
- 所有项目默认在 `策划后 / 每批正文后 / 交付前` 执行一次 `/复盘`
- 长线、海外、复杂题材或精品要求项目，默认每个关键里程碑都执行一次 `/复盘`
- 如果复盘发现问题较多，不要把问题清单原样扔进下一轮；默认再收一份 `改稿指令单-*.md`
- `改稿指令单-*.md` 只保留本轮必须改的 `3-5` 条动作，写清修改目标、影响范围、完成标准
- 如果不同来源的意见互相冲突，先在 `改稿指令单-*.md` 里收束成本轮取舍，再开始改稿
- 如果问题只集中在开场、尾钩、对峙场、首次亮相或单集局部，改稿指令单应优先指向 `局部返工`，不要默认整批重写
- 所有独立复盘都要按 [18-过程质检与独立复检.md](18-过程质检与独立复检.md) 写清复检方式、隔离证据、读取文件清单和用户复检指令

## 交付前全剧体验通读复盘

触发条件：
- 全部正文批次已完成并通过 `T14 剧本质检`
- 进入 `T15 全剧复核` 之前
- 用户要求"站在观众 / 编剧 / 导演 / 用户角度通读完整剧本"时

定位：
- 这是 `交付前复盘` 的必做子步骤，产物写入 `90-内部工作稿/复盘报告-交付前.md`
- 不替代 `T14 剧本质检`、`T15 独立全剧复核` 或 `T18 交付质检`
- 专门发现"机械硬阈值通过、批次独立质检也通过，但完整观看体验仍有问题"的情况
- 复检方式仍按本文件上文要求(`clean-context` / `subagent`)；运行环境适配见 [18-过程质检与独立复检.md](18-过程质检与独立复检.md) 的"运行环境适配"段

通读方法、三视角检查清单（用户 / 编剧 / 导演）、输出判断字段和不通过后的状态回退细则统一在 [13-质量检查表.md](13-质量检查表.md) 第三章；本文件不重复维护，避免两侧规则漂移。
