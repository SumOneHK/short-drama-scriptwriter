# 产物矩阵与创建时机

## 目的

这份文件是当前 skill 的 `单一产物口径`。

所有关于“某个文件是不是必须产物”的判断，都优先以这份矩阵为准。  
其他文件里的目录树、阶段细则、校验提示，都只能引用这份矩阵，不能各自发明一套说法。

特别说明：
- `目录树` 展示的是 `项目最终可能形态`，不是默认每次都要一次性生成的清单
- 稳定输出的关键不是“文件越多越好”，而是 `该出的稳定出，不该出的不要预创建`
- `项目状态.json` 和 `90-内部工作稿/质检检查点.md` 不属于可选工作稿，它们是确认立项并初始化项目后的流程门禁文件；确认前不创建
- 模板不是质量目标。正式产物模板只提供结构骨架，不能把填写规则、检查清单或方法论说明带进项目正文
- 如果一个模板主要用于判断、诊断、返工或阶段复检，它只能是工作稿或参考材料，不能进入正式交付

## 状态定义

- `必出`：默认每个项目都要有，没有就视为流程不完整
- `条件必出`：命中明确触发条件时必须生成；没命中时不要默认生成
- `可选工作稿`：只在当前轮创作或返工真的需要时才开
- `禁止预创建`：不能因为“以后可能用到”就提前空建

## 模板层级

| 层级 | 用途 | 处理原则 |
| --- | --- | --- |
| 正式产物模板 | 保证交付结构稳定 | 只保留必要字段和标题，不写长篇规则 |
| 工作稿模板 | 帮助当前轮写作、诊断、返工 | 按需启用，不进入正式交付 |
| 参考方法 | 提供判断标准和写作方法 | 只读不落盘，不复制到项目文件 |

## 四步创作链映射

用户按“创意 -> 梗概 -> 大纲/情节点 -> 剧本初稿”推进时，按下面映射执行，不必改文件名：

| 用户说法 | 对应任务 | 产物口径 |
| --- | --- | --- |
| 创意 | `T01-T04` + `T05-T07` 最小可写性校准 | `项目简报 / 锁题摘要 / 项目设定`，加最小人物压力链、世界规则和场景道具锚点 |
| 梗概 | `T08` 的 `故事梗概.md` | 用短文本讲清主戏、主角、压迫、反击和终局承诺 |
| 大纲 / 情节点 | `T08 故事大纲` + `T09 故事节拍表/阶段大纲` + `T10 分集大纲` | 先定全剧升级链，再钉关键翻面，最后拆到逐集情节点 |
| 剧本初稿 | `T13 第XXX集.md` | 场次执行稿；首批可先写 `1-2` 集代表样稿 |

执行原则：
- 用户已有创意时，不必从 T01 重做市场发散；补齐必要锁题信息后进入梗概
- 用户已有梗概时，不要回头扩写设定百科；只补会影响情节点拆解的最小人物、规则、场景锚点
- 用户已有大纲/情节点时，先做大纲质检；通过后进入剧本初稿
- `世界观设定 / 人物小传 / 场景道具锚点` 是创意可写性支撑，不是挡在梗概前面的百科任务

编号规则：
- `T05-T07` 属于同一轮故事基底包，可以在同一次策划工作里互相校准，但最终仍按 `世界观设定.md / 人物小传.md / 资产锚点或卡` 分别落盘
- `T08` 同轮生成 `故事梗概.md / 故事大纲.md`，不是两个完全割裂的阶段；梗概定戏，大纲定推进
- `T09` 同轮生成 `故事节拍表.md / 阶段大纲.md`；阶段大纲是否落盘按总集数和复杂度触发
- 同轮创作不等于提前铺满全部细节；先出最小可用版本，后续按大纲、节拍和分集需要回补
- `剧本圣经.md` 不是策划阶段产物；它只在 `T16` 把已通过质检的分项策划文件汇总成最终交付物

## `/立项`

| 文件 | 状态 | 触发条件 | 门禁关系 |
| --- | --- | --- | --- |
| `项目简报.md` | `必出` | 用户明确确认方向并进入 `T02` 后的所有项目 | `kickoff` 必查 |
| `锁题摘要.md` | `必出` | 用户明确确认方向并进入 `T02` 后的所有项目 | `kickoff` 必查 |
| `市场快照.md` | `条件归档` | 用户确认后仍要求归档市场判断，或平台/发行判断复杂 | 不单独放行；确认前不得创建 |
| `待选题材池.md` | `条件归档` | 用户确认后要求保留候选比较记录 | 不单独放行；确认前不得创建 |
| `故事理解.md` | `条件归档` | 同时满足：(1) 项目类型为 `改编 / 素材改写 / 长文本整理`；(2) 用户已确认进入 `T02` 立项落盘；(3) 用户明确要求落盘故事理解（不是默认归档） | `kickoff` 必查；只在确认归档时才与 `锁题摘要.md` 并列存在 |

稳定规则：
- 所有项目立项阶段都必须完成 `市场判断`
- `项目简报.md` 的职责是记录用户原始需求、确认信息、约束、默认假设和轻量市场判断；它不是最终创意决策表
- `锁题摘要.md` 的职责是记录最终锁定的创作方向、核心卖点、连载引擎、压迫来源和不可偏离边界；它不是原始需求备忘录
- 两份文件可以共享少量硬字段（平台、目标市场、规格、确认记录），但不要重复展开同一段市场分析、题材论证或卖点推导；需要引用时写“详见项目简报/锁题摘要对应段”
- 进入 `T02` 后，默认把 `市场判断` 轻量写入 `项目简报.md`，把影响最终创作方向的结论写入 `锁题摘要.md`
- 如果用户一开始就已经明确锁定 `目标市场 + 题材方向 + 项目边界`，`市场快照.md` 不是默认必出
- `市场快照.md / 待选题材池.md` 的职责是辅助决策；确认前只在对话中使用，确认后也不为完整性补建
- 只要用户仍在比较、要求继续调研或没有明确选择方向，流程停在对话态 `T01`；不得用推荐排序代替用户确认来生成任何项目文件

`故事理解.md` 与 `锁题摘要.md` 的分工（避免重复内容）：
- `故事理解.md` 的职责：把原始素材（小说 / 网文 / 长篇英文故事 / 真人真事 / 历史事件）整理成纯中文的"故事现状"，包括原作核心情节 / 主要人物 / 关键设定 / 改编可保留点 / 改编必须删除或重构点；它是**对原始素材的理解**，不是项目方向
- `锁题摘要.md` 的职责：在故事理解基础上，结合市场判断、目标受众、规格、本地化要求等，锁出**项目方向**；它是**改编后的项目锁题**，不是原作还原
- 两份文件**不允许重复内容**：故事理解只写"原作有什么"；锁题摘要只写"我们改成什么、为什么改"
- 改编项目的实操顺序：先写故事理解 → 用户确认要做哪一版改编方向 → 再写锁题摘要；锁题摘要里直接引用故事理解的关键段落，不重复描述原作
- 原创项目（非改编）：不写故事理解，只写锁题摘要

稳定规则：
- `分集大纲.md` 永远是正文前硬门槛
- `阶段大纲.md` 不再使用模糊的“按需”口径；默认按 `总集数` 判定是否进入正式产物
- `30` 集左右的国内项目默认已经需要阶段大纲，否则中段变奏、阶段回报和批次写作容易失控
- `世界观设定.md / 人物小传.md / 场景道具锚点` 要先形成最小可用基底，再进入 `故事梗概.md / 故事大纲.md`；但资产卡不因此默认必出
- `故事大纲.md / 故事节拍表.md / 阶段大纲.md` 稳定后，应回看世界规则、角色压力链、常驻场景和关键道具是否需要补校准
- `场景参考卡.md / 道具设定卡.md` 在策划层只在 `确实影响写作与制作判断` 时提前生成；否则默认推迟到交付增强包
- 如果 `项目设定.md / 阶段大纲.md / 世界观设定.md` 已经锁清关键场景、关键道具、核心视觉符号和高复用场景，标准剧本阶段不再为了目录完整单独补 `场景参考卡.md / 道具设定卡.md`

- 项目属于 `资产重 / 海外 / 强设定` 时，正式制作交付默认应使用 `production-enhanced`；如果只做标准交付，必须在独立质检里记录风险接受理由
- `03-交付/场景参考卡.md` 和 `03-交付/道具设定卡.md` 是最终版；不直接搬 `01-策划/` 的工作版
- `剧本圣经.md` 可以汇总场景 / 道具卡内容，但这只是单册阅读副本；独立的 `03-交付/场景参考卡.md` 与 `03-交付/道具设定卡.md` 仍是制作资产卡 source of truth
- 因此交付目录同时存在 `剧本圣经.md` 和独立场景 / 道具卡不算冗余；只有当项目不需要场景 / 道具资产交底时，才不生成独立卡，也不把策划工作版硬塞进交付层
