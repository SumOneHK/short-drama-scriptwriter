# 状态流转与阶段门禁

## 何时读

- 新项目启动时
- 需要判断能否进入下一步时
- 中途恢复项目现场时
- 同一步骤重复执行时
- 上游文档发生实质改动、需要回退门禁时

## 单一真相源

用户明确确认方向并进入 `T02`、项目初始化之后，全流程只认两个文件：
- 项目根目录 `项目状态.json`
- `90-内部工作稿/质检检查点.md`

确认前的 `/立项` 调研和候选讨论没有项目状态文件，不做门禁回写。

其中：
- `项目状态.json` 负责记录 `当前执行节点 / 项目粗阶段 / 产出进度 / 门禁状态 / 回退状态`
- `质检检查点.md` 负责记录 `每次独立质检证据 / 结论 / 阻塞问题 / 状态回写`

模板：
- [../assets/templates/23-项目状态模板.json](../assets/templates/23-项目状态模板.json)
- [../assets/templates/24-质检检查点模板.md](../assets/templates/24-质检检查点模板.md)

## 状态对象字段

### `projectPhase`

粗阶段，只允许：
- `立项`
- `策划`
- `剧本`
- `交付`

用途：
- 让恢复现场时先知道项目停在哪一大段
- 不替代 `currentStep`

### `currentStep`

细节点，只允许：
- `立项`
- `立项质检`
- `策划基底`
- `策划基底质检`
- `策划结构`
- `策划结构质检`
- `分集大纲`
- `大纲质检`
- `剧本批次`
- `剧本质检`
- `交付整合`
- `交付质检`

用途：
- 表示当前真正正在执行的节点
- `currentStep` 可以停在”质检节点”，这表示项目正在等待独立放行，而不是已经通过

### `lastCompletedPhase`

记录最近完成的关键里程碑，用来区分 `currentStep` 相同但实际进度不同的状态。只允许：
- `””`（空，项目还未到达任何关键里程碑）
- `立项已锁`
- `策划基底已通过`
- `策划结构已通过`
- `大纲已通过`
- `剧本批次已全部通过`
- `全剧复核已通过`
- `交付已通过`

`T15 全剧复核` 是这个字段的关键用途之一：
- T15 不占独立 `currentStep`（即使全剧复核完成，`currentStep` 仍可保持在 `剧本质检` 或 `交付整合`）
- T15 完成后必须把 `lastCompletedPhase` 写成 `全剧复核已通过`，并把 `scriptProgress.fullScriptReviewDone` 写成 `true`
- 没有 `全剧复核已通过` 之前，`qcStatus.delivery` 不允许写 `已通过`，即使所有 `scriptBatches[]` 已经全部通过

恢复现场时必须同时读 `currentStep` 和 `lastCompletedPhase`，不能只看 `currentStep` 判断进度。

### `scriptProgress.fullScriptReviewDone`

布尔字段，记录 `T15 全剧剧本复核` 是否已完成：
- `false`：未做或未通过全剧复核
- `true`：全剧复核已通过（必须配合 `lastCompletedPhase = 全剧复核已通过` 一起写）

写入规则：
- 全剧复核独立完成后才允许写 `true`，且必须在 `质检检查点.md` 留下证据化结论
- 任意一批 `scriptBatches[]` 状态从 `已通过` 回退为 `需修改` 时，本字段必须同步回退为 `false`，`lastCompletedPhase` 也回退到 `剧本批次已全部通过` 之前的状态
- `qcStatus.delivery` 写 `已通过` 的前置条件之一是本字段为 `true`

### `deliveryMode`

只允许：
- `planning-only`
- `standard`
- `production-enhanced`

含义：
- `planning-only`：还没进入交付整合，只做到策划或正文阶段
- `standard`：只做 `完整剧本总稿 / 剧本圣经 / 交付清单`
- `production-enhanced`：进入制作增强包，额外整理 `制作理解稿 / 制作交付参数表 / 场景卡 / 道具卡 / 海外制作版分场表`

规则：
- `currentStep` 进入 `交付整合 / 交付质检` 时，`deliveryMode` 不能继续停在 `planning-only`
- `03-交付/` 已经出现制作增强包文件时，`deliveryMode` 必须同步改为 `production-enhanced`
- 资产重、海外或强设定项目如果只做 `standard`，必须在独立质检里记录风险接受理由

### `qcStatus`

固定字段：
- `kickoff`
- `planningFoundation`
- `planningStructure`
- `outline`
- `scriptBatches`
- `delivery`

其中：
- 前五个普通门禁值统一使用 `未检查 / 已通过 / 需修改`
- `scriptBatches` 是批次数组，不是单一字符串

### `outlineProgress`

记录 `分集大纲` 的写作进度，不记录是否已放行。

最低字段：
- `completedRanges`
- `currentRange`
- `nextRange`
- `allOutlineGenerated`

语义：
- `completedRanges`：已经生成并整合进正式 `分集大纲.md` 的范围
- `allOutlineGenerated = true`：表示全量大纲已经写完并整合为单一正式文件
- 这仍然不等于 `qcStatus.outline = 已通过`

### `scriptProgress`

记录正文落盘进度，不记录是否已放行。

最低字段：
- `completedRanges`
- `currentRange`
- `nextRange`

语义：
- `completedRanges`：已经完成本批写作并可纳入批次质检/交付判断的范围
- `currentRange`：当前正在写的批次或代表样稿范围；这一范围内可以已经有部分 `第XXX集.md` 落盘
- 批次是否真正放行，看 `qcStatus.scriptBatches`
- 范围写法允许 `001-005` 或 `EP001-EP005`，同一项目内建议统一一种格式

### `changeControl`

这是严格状态机里专门负责 `上游改动回退` 的对象。

最低字段：
- `pendingRollback`
- `sourceStep`
- `sourceFiles`
- `affectedGates`
- `reason`
- `triggeredAt`
- `resolvedAt`

语义：
- `pendingRollback = true`：表示上游发生了实质改动，下游门禁已失效，正在回退修复中
- `affectedGates`：列出受影响的门禁，允许值为 `kickoff / planningFoundation / planningStructure / outline / scriptBatches / delivery`
- `resolvedAt` 为空，表示这次回退还没闭环

### `lastQcStep`

记录最近一次独立质检对应的节点，只允许：
- `立项质检`
- `策划基底质检`
- `策划结构质检`
- `大纲质检`
- `剧本质检`
- `交付质检`

### `lastIndependentReviewMode`

只允许：
- `clean-context`
- `subagent`（仅用户明确要求并行或代理复检时；普通隔离复检任务仍写 `clean-context`）

如果已经有独立质检通过，但这个字段还空着，说明状态维护不完整。

## 固定流程图

主流程固定为：

`对话态 T01 立项沟通 -> 用户确认并初始化项目 -> 立项 -> 立项质检 -> 策划基底 -> 策划基底质检 -> 策划结构 -> 策划结构质检 -> 分集大纲 -> 大纲质检 -> 剧本批次 -> 剧本质检 -> 交付整合 -> 交付质检`

说明：
- 每个 `创作节点` 后面都跟一个 `独立质检节点`
- 质检不通过时，不进入下一个创作节点
- 允许在同一节点重复执行；重复执行不新增平行主文件，只覆盖为当前最佳版本
- 任务编号写入阶段回复和 `质检检查点.md`；`currentStep` 只保留上面的少量门禁节点
- `策划基底` 对应 `T04-T09` 的可判断版本；`策划结构` 是进入 `T10 分集大纲` 前的结构复核节点，不强制新增主文件

## 阶段门禁表

| 当前要进入的节点 | 必须满足 |
| --- | --- |
| `立项` | 用户已明确确认方向和必要立项信息，项目已初始化 |
| `立项质检` | `00-立项/` 已写到可评审版本 |
| `策划基底` | `qcStatus.kickoff = 已通过` |
| `策划基底质检` | `T04-T09` 策划基底包已成形 |
| `策划结构` | `qcStatus.planningFoundation = 已通过` |
| `策划结构质检` | `T10` 前结构复核已完成 |
| `分集大纲` | `qcStatus.planningStructure = 已通过` |
| `大纲质检` | `分集大纲.md` 已全量完成并整合 |
| `剧本批次` | `qcStatus.outline = 已通过` |
| `剧本质检` | 当前批次正文已落盘 |
| `交付整合` | 目标交付范围的 `scriptBatches` 已全部 `已通过`，且 `T15 全剧复核` 已通过（`scriptProgress.fullScriptReviewDone = true` 且 `lastCompletedPhase = 全剧复核已通过`） |
| `交付质检` | 交付文件已生成，等待独立放行 |

## 重复执行规则

### 同一步骤重做

允许重做，但要遵守：
- 主文件继续使用同一文件名，不新建 `v2 / 最终版 / 修正版`
- `质检检查点.md` 追加新记录，不覆盖旧记录
- `项目状态.json` 只保留当前最新状态，不保留旧状态快照

### `分集大纲` 分批生成

允许边写边补，但状态上要严格区分：
- `outlineProgress.completedRanges`：已经完成哪些范围
- `qcStatus.outline`：全量大纲是否独立通过

放行规则：
- 可以边补 `001-020 / 021-040 / ...`
- 但只有 `allOutlineGenerated = true`
- 且全量 `分集大纲.md` 已通过独立 `大纲质检`
- 才能把 `qcStatus.outline` 写成 `已通过`

### `/剧本` 分批生成

允许批次写作，但状态上要严格区分：
- `scriptProgress.currentRange`：当前批次或代表样稿正在落盘
- `scriptProgress.completedRanges`：哪些正文批次已经完成，可进入或已经通过批次质检
- `qcStatus.scriptBatches`：哪些批次已经独立通过

一批正文写完，不等于下一批可写。  
只有当前批次已通过独立 `剧本质检`，才允许继续下一批。

## 失败后的状态处理

### 独立质检不通过

统一动作：
- 当前 `currentStep` 留在对应质检节点，或回退到该节点对应的创作节点
- 对应门禁写成 `需修改`
- 在 `质检检查点.md` 记录 `阻塞问题 / 必改项 / 下一步动作`
- 未修完前，不得推进到下一个节点

示例：
- `立项质检` 不通过：`qcStatus.kickoff = 需修改`，`currentStep` 回到 `立项`
- `策划基底质检` 不通过：`qcStatus.planningFoundation = 需修改`，`currentStep` 回到 `策划基底`
- `大纲质检` 不通过：`qcStatus.outline = 需修改`，`currentStep` 回到 `分集大纲`
- 某批 `剧本质检` 不通过：该批 `scriptBatches[].status = 需修改`，`currentStep` 回到 `剧本批次`
- `交付质检` 不通过：`qcStatus.delivery = 需修改`，`currentStep` 回到 `交付整合`

### 上游改动导致下游失效

这是 `changeControl` 发挥作用的场景。

触发条件示例：
- 改了 `人物小传` 的核心关系结构
- 改了 `故事大纲 / 故事节拍表` 的关键翻面
- 改了 `分集大纲` 的主目标、主回报、关键接口
- 改了某批正文的尾钩，导致后面批次承接失效

统一动作：
1. 在 `changeControl` 中写明：
   - `pendingRollback = true`
   - `sourceStep`
   - `sourceFiles`
   - `affectedGates`
   - `reason`
   - `triggeredAt`
2. 把受影响的下游门禁改成 `需修改`
3. 把 `currentStep` 回退到最上游的受影响创作节点
4. 修完并重新独立质检后：
   - 清空 `pendingRollback`
   - 写入 `resolvedAt`

## 回退映射

| 受影响门禁 | 应回退到的创作节点 |
| --- | --- |
| `kickoff` | `立项` |
| `planningFoundation` | `策划基底` |
| `planningStructure` | `策划结构` |
| `outline` | `分集大纲` |
| `scriptBatches` | `剧本批次` |
| `delivery` | `交付整合` |

最小回退原则：
- 只回退真正受影响的下游门禁
- 但不能因为怕麻烦而少回退

### `scriptBatches` 多批次回退细则

`scriptBatches` 是数组，多批可能同时受影响。当上游改动同时影响多个已通过批次时，按以下规则处理：

1. `currentStep` 必须停在 `剧本批次`，回退到所有受影响批次中**最早的那一批**
2. `scriptBatches[]` 数组里，所有受影响批次的 `status` 全部回写为 `需修改`；不允许只回退最近一批
3. 回退顺序按集号从小到大执行：先重写并复检最早批次 → 再依次往后处理；不允许跳过中间批次直接修最新一批
4. 如果上游改动还影响了未来未写批次的承接口径，把 `scriptProgress.nextRange` 回滚到最早受影响批次之前，并在 `changeControl.affectedGates` 里追加 `scriptBatches`
5. 多批次回退期间，禁止把 `qcStatus.delivery` 写成 `已通过`；即使整体交付包之前已经放行过

判断"受影响批次"的最小标准：
- 上游改动涉及主线、人物身份、关系结构、关键道具、关键翻面或核心规则之一
- 该批次正文里曾经引用、依赖或承接被改动的元素
- 该批次的 `开场承接 / 主回报 / 尾钩 / 下一集接口` 任意一项与改动后的版本对不上

只要满足上面任意一条，就要列入受影响批次清单。判断口径有歧义时，优先按"全部回退"处理，不按"只回退一批"处理。

## 恢复现场

恢复现场时按这个顺序检查：
1. `projectPhase`
2. `currentStep`
3. `qcStatus`
4. `outlineProgress`
5. `scriptProgress`
6. `changeControl`
7. `质检检查点.md` 最近一轮独立结论

禁止只看“现在写到哪”继续推进；必须先看“哪一层已经过门禁、有没有正在回退的失效门禁”。

## 状态维护最小动作

每执行完一个节点，都至少要更新这些状态：

### 完成创作节点后
- `currentStep` 切到下一个质检节点
- 更新对应的 `outlineProgress / scriptProgress`
- 如果是交付整合，按实际需求写 `deliveryMode`

### 完成独立质检后
- 写 `lastQcStep`
- 写 `lastIndependentReviewMode`
- 写对应 `qcStatus`
- 把结论同步写入 `质检检查点.md`
- 如果不通过，保持或回退到对应创作节点
- 如果通过，再进入下一创作节点
