# 过程质检与独立复检

## 核心原则

当前上下文里的自检不是门禁。

本 skill 的检查分三层：
- `内建自检`：写作当下自己查风险，只用于发现问题
- `确定性脚本计数`：用 `scripts/check_episode.py` 等脚本做硬阈值机械检查（叠加屏幕字字数、画内文字可读性、环境标签字、规则词频次、前置字段比例、有效行数），可作为”硬阈值是否越线”的可证否证据，但**不替代戏剧判断**，不能单独决定放行
- `独立复检`：用模型在隔离上下文里重新读稿做阶段放行；用户明确发出 `clean-context 质检 / 独立质检 / 做质检放行` 指令后，可由当前会话代为发起隔离复检任务，用户不必自己新开会话

只有 `独立复检` 才能决定是否进入下一步。

质检不是脚本化校验：
- 脚本只能做”机械可计数”的硬阈值检查（叠加屏幕字字数 / 画内文字可读性 / 环境标签字 / 规则词频次 / 前置字段比例 / 有效行数 / 文件存在 / 字段连贯），不能输出戏剧判断
- 不用关键词扫描、字段计数、自动评分或 `PASS` 输出替代读稿判断
- 脚本输出可作为独立复检的”硬阈值证据附件”，但戏剧判断（人物声音 / 对峙张力 / 钩子拉力 / 本地化深度）仍由模型读稿完成
- 复检必须引用具体内容证据，例如某个策划结论、某集大纲、某场动作链、某段对白或某个尾钩
- 如果只能说”格式完整”或”脚本全过”却说不出”哪里抓人、哪里撑得住、哪里不成立”，结论不得写 `通过`

## 允许的复检方式

允许两种：
- `clean-context`：默认方式。可以由用户在新会话中发起，也可以由当前会话在用户明确发出复检指令后，启动一个不继承当前写作上下文的隔离复检任务。两种方式都必须重新读取检查对象并写出证据化判断
- `subagent`：仅在用户明确要求并行/代理复检时作为复检方式记录；如果运行环境把普通隔离复检实现为子任务，但用户没有要求并行/代理，状态字段仍按 `clean-context` 记录，并在证据中写明 `reviewRunner`

### `clean-context` 硬性判定（三条全满足才算）

把以下三条当作硬门槛，任何一条不满足都不算 `clean-context`，结论只能写 `内建自检/建议结论`，不得写门禁放行：

1. `隔离上下文`：本次复检发生在用户新会话，或当前会话代为发起的无历史隔离复检任务里。隔离任务不得 fork 当前写作上下文，不得接收当前会话的判断结论；同一模型在当前上下文里“换个视角重读”不算隔离
2. `仅给正式源文件`：复检上下文里只读取正式产物（`项目简报.md / 锁题摘要.md / 项目设定.md / 世界观设定.md / 人物小传.md / 故事梗概.md / 故事大纲.md / 故事节拍表.md / 阶段大纲.md / 分集大纲.md / 第XXX集.md` 等）和 `references/` 中的方法文件；不带入 `90-内部工作稿/` 里的 `批次写作单 / 单集达标检查 / 试写样场 / 改稿指令单 / 复盘报告 / 项目状态.json` 等当前会话产生的判断和结论。复检结束后，当前会话可以读取状态文件做回写，但不能把状态文件作为通过依据
3. `显式复检指令`：用户明确发出复检指令（如 `请对当前 EPxxx-EPyyy 批次做剧本质检`、`对项目简报和锁题摘要做 clean-context 立项质检`），并指明检查对象的范围；自发性”再帮你检查一下”不算

实操判断：
- 如果你正在写正文同时被要求”再质检一遍”，不满足条件 1，只能做 `内建自检`
- 如果用户在同一会话里说”换一种思路重新检查”，但没有启动隔离复检任务，仍然不满足条件 1
- 如果用户在同一会话明确要求 `clean-context 质检`，默认由当前会话发起隔离复检任务；用户不需要自己新开会话
- 如果运行环境无法发起隔离复检任务，才让用户新开会话或只输出 `内建自检/建议结论`
- 如果新会话里一开始就给了”我已经判 P0、不要再质疑”这类前置结论，不满足条件 3，结论质量被污染，仍只能算 `内建自检`
- 模型不能自行声称”我现在切换到 clean-context 模式”；必须有用户明确复检指令，并且实际发生隔离

#### 可机器验证的 clean-context 证据

为了让”是否处于 clean-context”不依赖模型自我感知，每轮独立复检都把以下证据写入 `质检检查点.md`。证据可以由用户提供，也可以由当前会话在发起隔离复检任务时自动记录：

- 当前时间戳：用户给或取当前环境日期
- `reviewRunner`：`user-new-session` / `isolated-review-task` / `subagent`
- `contextIsolation`：说明是否无历史隔离、是否未 fork 当前写作上下文
- 读取文件清单：证明只给了正式源文件和必要 references，没有混入工作稿
- 用户的复检指令原句：证明 `显式复检指令` 条件
- `python3 scripts/check_episode.py` 的输出（如适用）：作为硬阈值越线情况的辅助证据

模型在质检检查点.md 写入：

```
- lastIndependentReviewMode: clean-context
- evidenceTimestamp: <用户提供时间或当前环境日期>
- reviewRunner: <user-new-session / isolated-review-task / subagent>
- contextIsolation: <无历史隔离；未 fork 当前写作上下文；只读正式源文件>
- evidenceFiles: <读取文件清单>
- evidenceInstruction: <用户原句>
- scriptCheckResult: <check_episode.py 的输出，如适用>
```

如果没有 `reviewRunner/contextIsolation/evidenceFiles/evidenceInstruction` 这些证据，模型不应自行写 `clean-context`；记录 `lastIndependentReviewMode: 内建自检`，等待补齐证据或重跑隔离复检。

可执行口径：
- 用户明确要求 `clean-context` 或阶段质检时，当前会话应优先代为发起隔离复检任务，而不是要求用户自己重开会话
- 如果当前运行环境无法形成真正的隔离复检，只能做 `内建自检`，不得把结论写成门禁放行
- 没有独立复检证据时，`currentStep` 保持在对应质检节点，`qcStatus` 不写 `已通过`
- 用户明确要求代理或并行复检时，才可把 `lastIndependentReviewMode` 写成 `subagent`
- 用户新会话复检，或当前会话代发起的隔离复检任务，均可把 `lastIndependentReviewMode` 写成 `clean-context`

不允许（自我放行红线，违反任意一条等同伪造门禁）：
- 在同一上下文里”我刚写完，我自己再评一遍，然后判通过”
- 模型自行判定”现在足够干净了”并把 `lastIndependentReviewMode` 写成 `clean-context`
- 把 `内建自检` 的结论直接回写成 `qcStatus=已通过`
- 运行质检脚本后把脚本结果当成阶段放行
- 用户没有明确发出质检/复检指令时，模型自行调用隔离任务或 subagent 跑质检并把结果当门禁

## 固定质检节点

| 质检节点 | 检查对象 | 通过后写回 | 不通过后动作 |
| --- | --- | --- | --- |
| `立项质检` | `项目简报 / 锁题摘要 / 条件立项文件` | `qcStatus.kickoff = 已通过` | `qcStatus.kickoff = 需修改`，回到 `立项` |
| `策划基底质检` | `T04-T09` 策划基底包 | `qcStatus.planningFoundation = 已通过` | `qcStatus.planningFoundation = 需修改`，回到 `策划基底` |
| `策划结构质检` | `T10` 前结构复核 | `qcStatus.planningStructure = 已通过` | `qcStatus.planningStructure = 需修改`，回到 `策划结构` |
| `大纲质检` | 全量 `分集大纲.md` | `qcStatus.outline = 已通过` | `qcStatus.outline = 需修改`，回到 `分集大纲` |
| `剧本质检` | 当前批次正文 | 对应 `scriptBatches[].status = 已通过` | 对应 `scriptBatches[].status = 需修改`，回到 `剧本批次` |
| `交付质检` | 当前交付包 | `qcStatus.delivery = 已通过` | `qcStatus.delivery = 需修改`，回到 `交付整合` |

说明：
- `立项质检` 只在用户明确确认方向、项目已初始化并完成 `项目简报 / 锁题摘要` 后执行；题材仍在比较或继续调研时停留在对话态 `T01`，不创建 `质检检查点.md`，不做放行。
- `分集大纲` 分批生成期间可以做过程检查，但这些都不算正文放行
- 正文放行只认“全量大纲独立质检通过”

## 质检结论等级

- `通过`：可进入下一步
- `需修改`：当前节点内容不成立，必须修后重检
- `阻塞`：存在 `P0`，禁止继续推进，且通常伴随回退

和问题等级的关系：
- 有 `P0`：必须是 `阻塞`
- 无 `P0` 但有重要 `P1`：通常是 `需修改`
- 允许带着 `P1` 推进时，必须在质检记录里写明理由

## `质检检查点.md` 记录要求

每轮独立复检至少写清：
- `检查点ID`
- `阶段`
- `复检方式`
- `检查对象`
- `检查范围`
- `结论`
- `P0 / P1 / P2`
- `阻塞问题`
- `是否允许进入下一步`
- `状态回写`
- `下一步动作`

每个 `P0/P1` 问题都必须附具体证据：
- 策划问题引用到文件和段落主题
- 大纲问题引用到集号、剧情拍点或场次承载表
- 正文问题引用到集号、场号、动作链、对白或尾钩
- 交付问题引用到交付文件和缺失/冲突的源文件

### 字段分档审查（MUST / SHOULD / MAY）

模板字段三档分级（详见 `SKILL.md` 模板字段分级原则）的质检要求：

- 任何 `MUST` 字段缺失或填成 `待定 / 暂略 / 后补`：自动判 `P0`，需修改
- 任何 `SHOULD` 字段被跳过但没在阶段回复或本检查点写明跳过原因：判 `P1`，需修改
- 任何 `MAY` 字段填了内容但项目未命中该字段触发条件（如非改编项目填了故事理解字段、IAA 项目填了 IAP 卡点字段）：判 `P2`，建议删字段——这是字段膨胀的反作弊信号
- 字段填得满 ≠ 完成度高；评估完成度只看 MUST 字段是否实质填写（即填了具体内容、不是抽象套话）

在 `质检检查点.md` 单独写一行：
```
fieldComplianceCheck:
  - missingMust: [字段A, 字段B]    # 若有
  - silentlySkippedShould: [字段C]  # 若有
  - filledIrrelevantMay: [字段D]    # 若有
```

## 独立复检后的状态回写

独立复检结束后，同步回写两处：
- `90-内部工作稿/质检检查点.md`
- `项目状态.json`

统一回写：
- `lastQcStep`
- `lastIndependentReviewMode`
- 对应 `qcStatus`
- `currentStep`

如果是批次剧本，还要回写：
- `qcStatus.scriptBatches[]`
- `scriptProgress`

如果是上游改动后的回退复检，还要回写：
- `changeControl`

## 复检失败后的处理

### 同节点失败

如果只是当前节点内容不成立：
- 不开启 `changeControl`
- 直接把对应门禁写成 `需修改`
- 回到该节点的创作步骤修

### 上游改动引发失败

如果这次复检发现“上游一改，下游全漂了”：
- 开启 `changeControl.pendingRollback = true`
- 写 `sourceFiles / affectedGates / reason`
- 把受影响门禁全部回退
- 下游不再继续写，先回到最上游受影响节点修复

## 重复复检规则

- 同一个节点可以多次复检
- `质检检查点.md` 只追加，不覆盖旧记录
- `项目状态.json` 只保留最新状态
- 如果同一批次连续两次没过，不要继续机械重检；先出 `改稿指令单` 收束本轮必须改的动作

## 与 `/复盘` 的关系

`/复盘` 是显式命令，但它不是独立于质检之外的第二套门禁系统。

在这个 skill 里：
- `策划后复盘` 对应 `T04-T09` 策划基底、`T10` 前结构复核或 `大纲质检` 前后的显式复查
- `批次复盘` 是 `剧本质检` 的配套检查动作
- `交付前复盘` 是 `交付质检` 的配套检查动作

### 复盘的角色定位（关键边界）

复盘是 `诊断工作稿`，不是 `门禁`：

- 复盘报告（`复盘报告-*.md`）只能列出问题、改稿优先级、下一批必须延续 / 禁止再犯什么
- 复盘报告里的 `结论 / 建议` 字段 **不能直接改写 `qcStatus`**
- 复盘报告 **不能单独决定** 阶段是否放行、是否进入下一节点
- 复盘的输出会影响下一批的约束（如下一批必改项 / 禁止重复 trope / 改稿指令单），但门禁本身仍然只看独立复检
- 即使复盘报告写"建议通过"，没有独立质检证据时 `qcStatus` 仍保持当前状态；即使复盘报告写"建议阻塞"，独立质检仍要走完独立流程作为正式判断
- 复盘和独立质检在同一个节点同时存在时，独立质检结论优先；二者结论冲突时，必须在 `质检检查点.md` 写明"复盘建议 X，独立质检判 Y，最终按独立质检处理"

结论只有一个：
- 要不要进入下一步，仍然只看独立复检写回后的状态
- 复盘的价值在 `下一批怎么改 / 怎么避免重复犯错`，不在 `本节点能不能放行`
